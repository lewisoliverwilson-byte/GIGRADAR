## Overview

Replaced mock localStorage authentication with a real user backend powered by Amazon Cognito and Amazon SES. New users must verify their email address with a 6-digit code before their account is activated. Existing sign-in flow uses Cognito SRP authentication with JWT session management.

All AWS infrastructure (Cognito User Pool, App Client) was provisioned directly via the AWS CLI — no manual Console steps required.

## Known Issues / Limitations

Carries forward all issues from v1.1, plus:

1. **No password reset flow** — there is no "forgot password" link. Users who forget their password have no self-service recovery path.

2. **No email change flow** — users cannot update their email address post-registration.

3. **Session expiry is silent** — if the Cognito JWT expires (default: 1 hour access token, 30 days refresh), the user is silently logged out on next page load with no explanation.

4. **Root account credentials in use** — the AWS CLI is authenticated as the root account. An IAM user with scoped permissions should be used for production.

5. **Single SES identity** — emails come from a personal Gmail address. A branded domain identity (e.g. `noreply@gigradar.co.uk`) would be more appropriate for production.
