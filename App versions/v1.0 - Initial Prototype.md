# GigRadar — Version 1.0: Initial Prototype

**Date:** 2026-04-03
**Deployment:** Initial build — static SPA served via Express on Node.js

---

## Overview

First working prototype of GigRadar, a UK gig discovery and tracking web app. Built as a fully client-side single-page application with mock data. No backend database or real authentication — all state is stored in `localStorage`.

---

## What Was Built

### Architecture
- **Express server** (`server.js`) serving a static `public/` directory
- **Hash-based SPA router** — no page reloads, all navigation via `window.location.hash`
- **Two JS files:** `data.js` (all mock data) and `app.js` (all views, routing, state)
- **One CSS file:** `main.css`
- AWS Amplify config (`amplify.yml`) included for deployment

### Pages & Views

#### Home (unauthenticated)
- Hero section with search bar and genre chip quick-filters
- Trending Artists grid (first 8 artists)
- Upcoming Gigs list (next 10 chronologically)

#### Home (authenticated + following artists)
- Personalised welcome greeting
- "Your Gigs" — upcoming gigs only from followed artists
- "Discover More" — artist grid excluding already-followed
- "More Upcoming" — general gig feed

#### Artist Page (`#/artist/:id`)
- Hero banner with gradient avatar, genre chips, listener count, country flag
- Follow / Unfollow button (persisted to localStorage)
- Three tabs:
  - **Upcoming Gigs** — gig cards with date badge, venue, ticket links
  - **Past Gigs & Setlists** — with collapsible setlist accordions where available
  - **About** — biography and stats (listeners, upcoming count, genres)

#### Browse / Genre Page (`#/genre/:name` or `#/genre/all`)
- Genre filter chips
- Artist grid filtered by selected genre

#### Calendar Page (`#/calendar`)
- Requires login — redirects to sign-in prompt if unauthenticated
- Monthly grid view, Monday-first
- Colour-coded dots on dates with gigs from followed artists
- Click a date to see gig cards for that day
- Previous/next month navigation

#### Search Page (`#/search?q=`)
- Matches artists by name, genre, and bio text
- Matches gigs by artist name, venue name, and venue city
- Separate sections for artist and gig results

### Features

- **Auth modal** — mock login and signup (no real backend). Derives display name from email. Persisted via localStorage.
- **Follow system** — follow/unfollow artists; persisted to localStorage
- **Notifications panel** — slide-out sidebar showing gig alerts; marks all as read on open; unread count badge on bell icon
- **Search autocomplete** — live dropdown in navbar with keyboard navigation (↑↓ Enter Esc)
- **Toast notifications** — for follow/unfollow, sign in/out events
- **Scroll-aware navbar** — adds `scrolled` class (shadow/blur) after 10px scroll
- **Gig cards** — date badge, venue, full date, per-seller ticket links with sold-out state
- **Artist cards** — gradient avatar, listener count, genre pills, upcoming gig count badge
- **Setlist accordions** — collapsible song lists on past gig entries
- **Keyboard accessibility** — artist cards navigable by keyboard; ARIA labels on interactive elements

## Known Issues / Limitations

1. **No real authentication** — login accepts any email/password combination; signup doesn't validate uniqueness. Purely cosmetic.

2. **Dates are hardcoded** — "today" is hardcoded as `2026-03-28` in multiple places in `app.js` and `data.js` rather than using `new Date()`. This means the "upcoming" vs "past" split will become incorrect over time without updates.

3. **Ticket links are placeholder `#`** — all ticket URLs point to `#` rather than real ticketing sites. Clicking them does nothing useful.

4. **No venue pages** — venue names in gig cards are not clickable/linked; there is no venue detail page showing all gigs at that venue.

5. **No real notifications** — the notification panel shows static mock data from `data.js`; notifications do not dynamically update when a user follows a new artist.

6. **Calendar only shows followed-artist gigs** — no way to browse all gigs in calendar view without following an artist first.

7. **No pagination** — artist and gig lists are capped by slicing arrays; no load-more or infinite scroll.

8. **No offline support / PWA** — no service worker or manifest file.

9. **`artistAvatar` has a duplicate key** — `#f59e0b` appears twice in the colour map in `app.js` (line ~110); the second entry silently overwrites the first.

10. **No mobile-optimised navbar search** — the search bar in the navbar may overflow on small screens; no collapsed hamburger menu.
