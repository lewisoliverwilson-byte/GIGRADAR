# v3.5 — Venue Crawl IAM Fix

**Date:** 2026-04-18

## Summary

Fixed a silent permission failure that prevented the continuous venue website crawler from running in Lambda. Venue crawl now fully operational — 201 gigs found from the first 50 venues.

## What Changed

### Lambda IAM Policy — venues table added
- **Problem:** `gigradar-lambda-role` only had DynamoDB access to `gigradar-artists` and `gigradar-gigs`. The `gigradar-venues` table was never included.
- **Symptom:** `fetchVenueBatch` DynamoDB scan was swallowing the `AccessDeniedException` via `.catch(() => ({ Items: [] }))`, returning 0 venues silently. Logs showed "Venue crawl done" instantly with no "Venue crawl: N venues..." line preceding it.
- **Fix:** Added `gigradar-venues` and `gigradar-venues/*` to the Lambda inline IAM policy. Also changed the `.catch` to log the error message so future permission issues surface immediately.
- **Result:** Venue crawl works — **201 gigs from 50 venues** on first run.

### Error Logging
- Changed silent `.catch(() => ({ Items: [] }))` to `.catch(e => { console.error('Venue scan error:', e.message); return { Items: [] }; })` in `fetchVenueBatch`.
